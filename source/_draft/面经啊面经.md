#### 如何减轻服务端压力

1. **缓存：** 使用缓存可以减少服务器负载，提高响应速度。缓存可以包括前端缓存（如浏览器缓存）和后端缓存（如CDN、Redis或Memcached）。通过缓存常用数据、页面片段或完整页面，可以减少服务器端的请求和处理压力。
    
2. **负载均衡：** 使用负载均衡技术将流量分发到多个服务器实例上。负载均衡器可以根据不同的算法（如轮询、最小连接数等）将请求分配到可用的服务器，从而降低单个服务器的压力。
    
3. **服务端缓存：** 在服务端缓存经常访问的数据或计算结果，避免重复的数据库查询或复杂的计算过程。这可以通过内存缓存（如Redis或Memcached）或分布式缓存（如Redis Cluster）来实现。
    
4. **异步处理：** 将耗时的任务或IO操作转移到后台线程或独立的服务中异步处理，以避免阻塞服务器主线程。例如，可以使用消息队列（如RabbitMQ或Kafka）来处理异步任务。
    
5. **使用CDN：** 使用内容分发网络（CDN）可以将静态资源（如图片、CSS和JavaScript文件）缓存到全球分布的节点上，从而减少服务器的负载并提高用户访问速度。
    
6. **合理设计API：** 设计高效的API接口，避免在单个请求中返回过多的数据或嵌套的数据结构。可以使用分页、筛选和懒加载等技术来减少数据传输量和处理时间。
    
7. **优化数据库：** 使用合适的数据库索引、查询优化和数据分区等技术来提高数据库的性能和扩展性，从而减轻服务器端的数据库压力。
    
8. **监控和调优：** 定期监控服务器的性能指标（如CPU、内存、网络和磁盘利用率），并根据需求对系统进行调优和优化，以确保服务器能够稳定高效地运行。

#### 线上出问题怎么排查，关注哪些指标，如何快速恢复

排查步骤：

- 先确认问题影响范围，受影响的服务；
- 查看服务日志，定为错误信息；
- 查看监控系统中的性能指标，如CPU、内存、磁盘和网络利用率，以及请求量、响应时间等指标。这些指标可以帮助确定是否存在性能瓶颈或异常情况；
- 如果问题与最近的代码变更相关，可以考虑回滚这些变更以暂时恢复正常运行；

关注指标：

- **响应时间：** 监控请求的响应时间，确保它们在正常范围内。较长的响应时间可能表示性能问题或资源不足。
- **错误率：** 关注错误率，包括HTTP错误码、异常堆栈和业务逻辑错误。较高的错误率可能表示代码bug或异常情况。
- **资源利用率：** 监控服务器资源的利用率，如CPU、内存、磁盘和网络。高资源利用率可能导致性能下降或系统崩溃。
- **流量：** 关注系统的请求量和流量模式，以及是否存在异常的请求行为或DDoS攻击。

2. **提升静态内容：** 编译器会将识别到的静态内容提升到渲染函数的顶部，以避免在每次渲染时都重新计算和生成这些静态内容。
    
3. **生成优化后的渲染函数：** 编译器根据提升后的静态内容和动态内容生成优化后的渲染函数。这个渲染函数只会在组件初始化时执行一次，避免了在每次重新渲染时都重新计算静态内容。

#### Webpack 的执行流程是什么

1. **解析：** Webpack 会从入口文件开始，递归地解析代码中的模块依赖关系。它会分析代码中的 import、require 等语句，以确定模块之间的依赖关系。
    
2. **加载器（Loader）：** 当 Webpack 遇到非 JavaScript 文件（如 CSS、图片等），它会使用相应的加载器来处理这些文件。加载器可以将这些文件转换为 JavaScript 模块，以便在打包过程中统一管理。
    
3. **转换：** 在加载器处理后，Webpack 将会对每个模块进行转换。这可能包括通过 Babel 进行 ES6+ 语法转换、通过 TypeScript 编译器进行 TypeScript 转换等。
    
4. **分块（Chunking）：** 在确定了所有模块及其依赖关系后，Webpack 将会根据配置的分块规则将模块分割成不同的块（Chunks）。这有助于优化加载时间，例如将第三方库、业务逻辑代码等分开打包。
    
5. **合并：** Webpack 将分块后的模块合并成一个或多个 bundle 文件。这些文件通常是最终部署到生产环境中的文件，用于在浏览器中运行应用程序。
    
6. **输出：** 最后，Webpack 将生成的 bundle 文件输出到指定的目录中。这些文件可以通过 HTTP 服务器提供给浏览器加载。

#### tree-shake 如何实现

1. **静态分析：** Tree shaking 工具会对代码进行静态分析，识别出模块之间的依赖关系以及模块中导出的内容。这个分析过程是在编译时进行的，不会执行代码，而是仅仅根据代码的结构和依赖关系进行分析。
    
2. **标记未使用的代码：** 在静态分析的过程中，tree shaking 工具会标记哪些模块和模块中的导出内容是被使用的，哪些是未被使用的。这种标记可以是通过各种方式实现的，比如使用 AST（抽象语法树）来遍历代码，并记录哪些导出被引用了。
    
3. **删除未使用的代码：** 一旦标记出未使用的代码，tree shaking 工具就会从最终的打包结果中删除这些代码。这意味着未使用的模块以及模块中未被引用的导出将不会包含在最终的打包文件中，从而减小了文件的大小。
    
4. **依赖图优化：** 除了删除未使用的代码外，一些 tree shaking 工具还可以进行依赖图的优化，以进一步减小打包后文件的大小。这可能包括合并相似的模块、移除无用的依赖等。

#### 使用 const 声明一个对象，对象可改吗，为什么
